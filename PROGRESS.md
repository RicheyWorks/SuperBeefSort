- **WikiSort** — `WikiSortStrategy` (`merge.wiki`): O(n log n) stable in-place block merge, the successor to `merge.inplace` (O(n log² n)). Bottom-up + iterative (O(1) stack). Each merge takes a block-merge fast path on all-distinct regions — pull the √-sized largest-element internal buffer to the far right, align the A/B seam to the block grid, selection-sort blocks by head, repair seams with buffer-backed local merges, then insertion-sort the buffer in place (no re-merge: it already holds the largest values) — and falls back to the stable rotation merge for short or duplicate-bearing runs. Registered in `BuiltinStrategyProvider`, added to `DifferentialTest`, covered by `WikiSortTest`. Verified: matches the JDK reference across random + all pathological shapes, stable across key densities, and move count grows as O(n log n) (≈25% fewer moves than `merge.inplace` at n=1M, gap widening with n), at the cost of a larger comparison constant (~3.6 n log₂ n vs 1.3 — the usual block-merge trade).
- **WikiSort benchmarking & curve report** — `merge.wiki` and `merge.inplace` added to the JMH `SortStrategyBenchmark` (wall-clock, three shapes at n=100k). New `MoveCurveReport` (`./gradlew moveCurve`) prints the metered move/comparison growth curve for merge / merge.inplace / merge.wiki normalised to n·log₂ n — `merge.inplace`'s moves climb ~4→9 (1k→1M) while `merge.wiki` flattens ~4.6→6.8, making the O(n log n) vs O(n log² n) difference reproducible.
- **WikiSort selector routing** — `RuleBasedStrategySelector` picks `merge.wiki` under the `STABLE` policy for very large (≥ 2²¹ ≈ 2M elements) mostly-distinct inputs (stable, O(1) aux, O(n log n) moves), keeping plain merge sort for smaller or duplicate-heavy inputs (where WikiSort would only fall back to a rotation merge). Covered by `RuleBasedSelectorTest`; the 50k-distinct profile in `BanditSelectorTest` stays on `merge` (below the size threshold).
- **Cost-model + bandit STABLE routing** — `CostModelStrategySelector`'s STABLE branch now mirrors the rule-based one (large mostly-distinct → `merge.wiki`, else merge); `BanditStrategySelector` inherits it for free by delegating deterministic policies to the rule-based base. Under `SMART` all three keep optimizing comparisons+moves, where plain merge dominates WikiSort (fewer of both), so WikiSort is correctly offered only when stability is requested. Covered by `CostModelSelectorTest` + `BanditSelectorTest`.
- **JMH wall-clock + threshold tuning** — measured all comparison sorts at n=100k (JDK 21). WikiSort is the *slowest* (~32 ms random, ~2.4× plain merge): wall-clock is comparison-dominated and WikiSort has the highest comparison constant. It is dominated by plain merge on every `SortResult` metric (comparisons, moves, time) — its only edge is O(1) memory. Accordingly raised `WIKI_MIN_SIZE` 2¹⁷ → 2²¹ (~2M, ~16 MB scratch) in both selectors, so `STABLE` auto-selects WikiSort only where merge's O(n) aux is genuinely prohibitive.
- **WikiSort comparison-constant cut (threaded distinctness)** — the all-distinct gate is no longer recomputed from scratch at every merge. `WikiSortStrategy` threads a monotone `trustDistinct` flag (seeded by a one-time internal-duplicate scan of the insertion-sorted base runs): while it holds, every run produced so far is strictly increasing, so a merge decides distinctness with a single *cross-run-equal* walk (`crossRunEqual`, one comparison per element) instead of the two-comparison `fullyDistinct` (which also re-verifies within-run order). The first duplicate seen anywhere clears the flag, after which the exact, self-contained `fullyDistinct` gate is used — so the block-vs-rotation choice (hence the output, the move count, and stability) is provably identical to the old per-merge gate on every input; only the comparison count falls. Measured via `moveCurve` on shuffled distinct permutations: `merge.wiki`'s cmp/(n·log₂n) drops ~16–19% (3.21→2.71 at n=1k; 3.96→3.21 at n=1M) while moves/(n·log₂n) stay byte-identical (4.64 / 5.81 / 6.45 / 6.79). Verified green on `WikiSortTest` + `DifferentialTest` (matches the JDK reference across every pathological + duplicate-heavy shape, stable across key densities, comparisons within the O(n log n) bound).
- **Auxiliary-memory capability + memory-budgeted STABLE routing** — `StrategyCapabilities` now carries an `AuxMemory` growth class (`CONSTANT` / `LOGARITHMIC` / `LINEAR`) with `estimatedBytes(n)` (8 B/ref: `CONSTANT`→0, `LOGARITHMIC`→8·⌈log₂n⌉, `LINEAR`→8·n). Every strategy declares its class: in-place sorts default to `CONSTANT` (insertion, heap, network, `merge.inplace`, `merge.wiki`); introsort and quicksort are `LOGARITHMIC` (recursion stack); merge, counting, LSD/MSD radix, learned, and TimSort are `LINEAR` (scratch / count / bucket arrays). Both `RuleBasedStrategySelector` and `CostModelStrategySelector` now derive the WikiSort `STABLE` crossover from an explicit 16 MB auxiliary-memory budget — `AuxMemory.LINEAR.estimatedBytes(size) >= AUX_MEMORY_BUDGET_BYTES` — instead of the magic `WIKI_MIN_SIZE = 2²¹`. This is byte-for-byte equivalent to the old gate (`8 B · 2²¹ == 16 MB`, and `>=` keeps the exact-`2²¹` boundary on WikiSort), so the magic number becomes a principled, self-documenting budget with no behavior change. Pinned by a new `StrategyCapabilitiesTest` (per-class `estimatedBytes` + builder default `CONSTANT`) and exact-boundary crossover cases added to `RuleBasedSelectorTest` / `CostModelSelectorTest` (2²¹ → `merge.wiki`, 2²¹−1 → `merge`).
- **Memory-budget filter for SMART (`CostModelStrategySelector`)** — the `AuxMemory` capability now shapes the `SMART` objective too, not just `STABLE`. `CostModelStrategySelector` gained an optional `smartAuxBudgetBytes` cap (new constructor; default `Long.MAX_VALUE` = unlimited, so the no-arg selector is byte-for-byte the old behaviour). Under a set budget, any candidate whose declared `AuxMemory.estimatedBytes(n)` exceeds it is excluded via `withinSmartBudget(...)` (reads capabilities from the registry, short-circuits when unlimited so the default path stays allocation-free), so memory pressure degrades selection to the in-place introsort / insertion instead of the LINEAR-aux counting / radix / learned / TimSort. The static budget was renamed `AUX_MEMORY_BUDGET_BYTES` → `STABLE_WIKI_CROSSOVER_BYTES` to mark it as the *separate* fixed 16 MB merge→WikiSort crossover used only by `STABLE`. This reframes the original "prefer WikiSort under SMART" idea, which doesn't hold up: WikiSort is dominated by introsort on comparisons / moves / time and `SMART` doesn't value stability, so the coherent feature is a general memory-aware objective (WikiSort remains a `STABLE`-only specialist). Covered by `CostModelSelectorTest` (default → counting; 1 MB cap → introsort; budget == 8n inclusive → counting; non-positive budget rejected) and verified numerically. `RuleBasedStrategySelector` is unchanged; the bandit's SMART path gets the same budget in the next item.
- **Bandit honours the SMART aux-memory budget** — `BanditStrategySelector` now accepts the same optional budget (new `BanditStrategySelector(long)` convenience plus a full 4-arg constructor; the no-arg and 3-arg constructors delegate with `Long.MAX_VALUE` = unlimited, so default behaviour is unchanged). Its `candidateArms` drops any arm whose declared `AuxMemory.estimatedBytes(n)` exceeds the budget — applied after capability gating and before the empty→introsort fallback — so under memory pressure the UCB search explores only in-place sorts: heap (`CONSTANT`) and intro / quick (`LOGARITHMIC`) survive while `LINEAR` merge / TimSort / counting / radix / learned drop out. Non-SMART policies still delegate to the rule-based base unchanged. Covered by `BanditSelectorTest` (a 256 KB cap at n=50k excludes the LINEAR comparison arms, and the LINEAR integer arms, even when they are made to look cheap; the default unlimited bandit still learns a LINEAR winner; non-positive budget rejected); arm filtering verified numerically.
- **`BeefSort.maxAuxMemory(long)` facade shortcut** — callers can cap SMART auxiliary memory without constructing a selector by hand: `BeefSort.…​maxAuxMemory(bytes)` installs a budgeted `CostModelStrategySelector(bytes)` through a new `resolveSelector()`, wired into both the `run()` / `feedInto()` paths and `adaptiveStream()`. It is mutually exclusive with an explicit `selector(...)` — a budget already installs a selector, so setting both throws `IllegalStateException` (to budget a different selector, e.g. the bandit, construct it with the budget and pass it to `selector(...)`) — and validates `bytes > 0`. SMART-only, mirroring the selector semantics; `STABLE` keeps its own crossover. Covered by `BeefSortBudgetTest` (256 KB cap → introsort; 1 GB cap → counting; mutual-exclusion + non-positive throws; output verified sorted under a budget).
- **Duplicate-tolerant `merge.wiki` (faithful WikiSort port)** — `WikiSortStrategy` is now a faithful, no-cache port of BonzaiThePenguin's WikiSort (the Kim&nbsp;&amp;&nbsp;Kutzner block merge), replacing the previous distinct-only fast path + rotation fallback. It handles duplicate keys **natively**, so near-distinct and duplicate-bearing inputs keep O(n&nbsp;log&nbsp;n) moves instead of degrading to the rotation merge's O(n&nbsp;log²&nbsp;n). Per merge level it pulls two √n *unique-value* buffers (one tags the A blocks for stable selection, one is swap scratch), rolls the A blocks through B dropping the minimum-headed block — ties broken by the tag ⇒ original order ⇒ stability — with buffer-backed seam merges, then redistributes; regions too low-cardinality to yield buffers fall back to the strictly-in-place rotation merge (`mergeInPlace`). ID, capabilities (stable + in-place + O(1) aux), and registration are unchanged, so the selectors/engine are untouched. **ADR correction:** the earlier per-merge `blockMergeDup` framing was incompatible with the proven algorithm — WikiSort's buffers are *per-level*, and stability needs the asymmetric A-block roll, not a symmetric block-sort with origin-tagged seams (see the updated `docs/adr-wikisort-duplicate-block-merge.md`). Verified by porting to Python and fuzzing the algorithm to stability (~7k cases, 0 failures: distinct / near-distinct / duplicate-heavy / all-equal / sawtooth / organ-pipe / reverse), then transcribing the Java back into a Python mirror and re-fuzzing (~7.4k cases, 0 failures) to catch translation drift; the `SortBuffer` API was cross-checked. Covered by `WikiSortTest` (incl. new dense-duplicate and near-distinct stability-at-scale cases) + `DifferentialTest`. The compile/test gate runs on the developer machine — the dev sandbox has no JDK&nbsp;17 / CSRBT, so the Java was verified by faithful-port equivalence rather than executed here.
- **Unified aux-memory budget across SMART and STABLE** — a configured `smartAuxBudgetBytes` on `CostModelStrategySelector` now also drives the STABLE merge→WikiSort crossover, not just SMART candidate filtering: `stable()` uses the budget as the crossover when one is set, falling back to the fixed 16 MB `STABLE_WIKI_CROSSOVER_BYTES` when unset — so the no-arg selector is unchanged. One budget governs both policies, and a memory-constrained run prefers the O(1)-aux WikiSort sooner under STABLE. Covered by `CostModelSelectorTest` (a 1 MB budget routes a 200k mostly-distinct STABLE input to `merge.wiki`, where the default 16 MB crossover keeps it on `merge`).
- **Measured peak auxiliary memory** — strategies now self-report their scratch allocation through a new `SortBuffer.recordAux(bytes)` meter (peak-tracked, mirroring the comparison/move metering), surfaced as `SortResult.peakAuxBytes` and shown in `SortReport`. The six allocating strategies report their actual footprint — merge ≈ 16n (working copy + scratch), TimSort ≈ 8n (reference copy), counting ≈ 16n + 4·range, LSD radix ≈ 32n + 4·radix, MSD radix ≈ 16n + 4·buckets, learned ≈ 28n — while the in-place sorts (insertion, heap, network, quick, intro, `merge.inplace`, `merge.wiki`) report 0. This is a real, per-strategy signal *finer than* the static `AuxMemory` class: it distinguishes a 4-array LSD radix from a single-buffer merge even though both are coarsely `LINEAR`, so a learning selector's cost function (`BanditStrategySelector` takes a pluggable `ToDoubleFunction<SortResult>`) can tune on observed memory instead of the estimate. Covered by `MeasuredAuxTest` (merge reports exactly 16n; in-place heap reports 0; the value flows through `SortRunResult.sortMetrics().peakAuxBytes()`).
- **Memory-weighted bandit cost function** — `BanditStrategySelector.costWithMemory(bytesWeight)` returns a `ToDoubleFunction<SortResult>` = comparisons + moves + `bytesWeight`·`peakAuxBytes`, so the self-tuning bandit can learn from the measured aux (above): pass it as the `cost` argument. With a heavy weight the UCB converges away from scratch-heavy sorts (merge / TimSort / radix) toward the zero-aux in-place sorts; `bytesWeight = 0` reproduces the default comparisons + moves objective exactly. This closes the loop — capability → budgets → measured aux → learning that *uses* it. Covered by `BanditSelectorTest` (heavy weight + equal comparisons/moves but LINEAR-aux sorts reporting 16n / 8n scratch ⇒ an in-place winner; zero weight ⇒ identical to the default objective).
- **CSRBT integration audit + post-feed observability (Gap 3)** — `docs/adr-csrbt-integration-deepening.md` audits the SBS↔CSRBT integration against the *actual* code (correcting an external review: `StrategyAdvisor`, `EnsembleTargetFactory`, `ProfileGuidedScorer`, the `WorkloadAdaptation`/`MorphController` wiring, and `buildOrderedSet`/`buildEnsemble`/`buildCoOptimized` are all already shipped — ~90% of the proposed "full CSRBT integration"), and scopes the five genuine gaps. Closed the highest-priority one (post-feed observability): `WorkloadAdaptation` now accumulates a `csrbt/AdaptationReport{evaluations, morphs, currentStrategy}` from the `MorphController.MorphResult` stream that `maybeAdapt()` already returns — no CSRBT `TreeEventListener` needed — making the §5 *"born right ⇒ zero morphs"* success metric assertable via `adaptationReport().held()`. `WorkloadAdaptationTest` now pins that a READ_HEAVY-born (AVL) tree HOLDs (zero morphs) under a matching read workload. The remaining gaps (member-mix `EnsembleSpec` + persistent member, order-statistics surfacing, measured-signal prior) are tracked in the ADR.
- **CSRBT ensemble adaptation / promotion (Gap 1)** — `csrbt/EnsembleAdaptation<K>` (+ `EnsembleAdaptationReport`) is the ensemble analog of `WorkloadAdaptation`: it composes CSRBT's `EnsembleController`, routes live `add`/`remove`/`contains` through it (feeding the workload monitor), and exposes `maybePromote()` — the **O(1) read-path primary swap** to the member matching live traffic, no rebuild — plus `checkHealth()` (failover / quarantine / heal). `BeefSort.buildAdaptiveEnsemble(MorphPolicy)` builds the profile-composed ensemble and returns the wired adapter. Closes the §4 promotion gap the integration design specified but SBS hadn't implemented; built against CSRBT's real `ensemble`/`control` API (folder now mounted). `EnsembleAdaptationTest` mirrors CSRBT's own `EnsembleControllerTest.skewedReadsPromoteSplayWithoutRebuild` through the adapter (RB+AVL+Splay; a skewed hot-key read stream promotes the read path to Splay **exactly once**). The default 2-member factory mix (access-advised primary + RedBlack replica) often lacks the diversity to promote — that's Gap 2.
