- **WikiSort** — `WikiSortStrategy` (`merge.wiki`): O(n log n) stable in-place block merge, the successor to `merge.inplace` (O(n log² n)). Bottom-up + iterative (O(1) stack). Each merge takes a block-merge fast path on all-distinct regions — pull the √-sized largest-element internal buffer to the far right, align the A/B seam to the block grid, selection-sort blocks by head, repair seams with buffer-backed local merges, then insertion-sort the buffer in place (no re-merge: it already holds the largest values) — and falls back to the stable rotation merge for short or duplicate-bearing runs. Registered in `BuiltinStrategyProvider`, added to `DifferentialTest`, covered by `WikiSortTest`. Verified: matches the JDK reference across random + all pathological shapes, stable across key densities, and move count grows as O(n log n) (≈25% fewer moves than `merge.inplace` at n=1M, gap widening with n), at the cost of a larger comparison constant (~3.6 n log₂ n vs 1.3 — the usual block-merge trade).

- **JMH wall-clock + threshold tuning** — measured all comparison sorts at n=100k (JDK 21). WikiSort is the *slowest* (~32 ms random, ~2.4× plain merge): wall-clock is comparison-dominated and WikiSort has the highest comparison constant. It is dominated by plain merge on every `SortResult` metric (comparisons, moves, time) — its only edge is O(1) memory. Accordingly raised `WIKI_MIN_SIZE` 2¹⁷ → 2²¹ (~2M, ~16 MB scratch) in both selectors, so `STABLE` auto-selects WikiSort only where merge's O(n) aux is genuinely prohibitive.
- **WikiSort comparison-constant cut (threaded distinctness)** — the all-distinct gate is no longer recomputed from scratch at every merge. `WikiSortStrategy` threads a monotone `trustDistinct` flag (seeded by a one-time internal-duplicate scan of the insertion-sorted base runs): while it holds, every run produced so far is strictly increasing, so a merge decides distinctness with a single *cross-run-equal* walk (`crossRunEqual`, one comparison per element) instead of the two-comparison `fullyDistinct` (which also re-verifies within-run order). The first duplicate seen anywhere clears the flag, after which the exact, self-contained `fullyDistinct` gate is used — so the block-vs-rotation choice (hence the output, the move count, and stability) is provably identical to the old per-merge gate on every input; only the comparison count falls. Measured via `moveCurve` on shuffled distinct permutations: `merge.wiki`'s cmp/(n·log₂n) drops ~16–19% (3.21→2.71 at n=1k; 3.96→3.21 at n=1M) while moves/(n·log₂n) stay byte-identical (4.64 / 5.81 / 6.45 / 6.79). Verified green on `WikiSortTest` + `DifferentialTest` (matches the JDK reference across every pathological + duplicate-heavy shape, stable across key densities, comparisons within the O(n log n) bound).
- **Auxiliary-memory capability + memory-budgeted STABLE routing** — `StrategyCapabilities` now carries an `AuxMemory` growth class (`CONSTANT` / `LOGARITHMIC` / `LINEAR`) with `estimatedBytes(n)` (8 B/ref: `CONSTANT`→0, `LOGARITHMIC`→8·⌈log₂n⌉, `LINEAR`→8·n). Every strategy declares its class: in-place sorts default to `CONSTANT` (insertion, heap, network, `merge.inplace`, `merge.wiki`); introsort and quicksort are `LOGARITHMIC` (recursion stack); merge, counting, LSD/MSD radix, learned, and TimSort are `LINEAR` (scratch / count / bucket arrays). Both `RuleBasedStrategySelector` and `CostModelStrategySelector` now derive the WikiSort `STABLE` crossover from an explicit 16 MB auxiliary-memory budget — `AuxMemory.LINEAR.estimatedBytes(size) >= AUX_MEMORY_BUDGET_BYTES` — instead of the magic `WIKI_MIN_SIZE = 2²¹`. This is byte-for-byte equivalent to the old gate (`8 B · 2²¹ == 16 MB`, and `>=` keeps the exact-`2²¹` boundary on WikiSort), so the magic number becomes a principled, self-documenting budget with no behavior change. Pinned by a new `StrategyCapabilitiesTest` (per-class `estimatedBytes` + builder default `CONSTANT`) and exact-boundary crossover cases added to `RuleBasedSelectorTest` / `CostModelSelectorTest` (2²¹ → `merge.wiki`, 2²¹−1 → `merge`).
- **Memory-budget filter for SMART (`CostModelStrategySelector`)** — the `AuxMemory` capability now shapes the `SMART` objective too, not just `STABLE`. `CostModelStrategySelector` gained an optional `smartAuxBudgetBytes` cap (new constructor; default `Long.MAX_VALUE` = unlimited, so the no-arg selector is byte-for-byte the old behaviour). Under a set budget, any candidate whose declared `AuxMemory.estimatedBytes(n)` exceeds it is excluded via `withinSmartBudget(...)` (reads capabilities from the registry, short-circuits when unlimited so the default path stays allocation-free), so memory pressure degrades selection to the in-place introsort / insertion instead of the LINEAR-aux counting / radix / learned / TimSort. The static budget was renamed `AUX_MEMORY_BUDGET_BYTES` → `STABLE_WIKI_CROSSOVER_BYTES` to mark it as the *separate* fixed 16 MB merge→WikiSort crossover used only by `STABLE`. This reframes the original "prefer WikiSort under SMART" idea, which doesn't hold up: WikiSort is dominated by introsort on comparisons / moves / time and `SMART` doesn't value stability, so the coherent feature is a general memory-aware objective (WikiSort remains a `STABLE`-only specialist). Covered by `CostModelSelectorTest` (default → counting; 1 MB cap → introsort; budget == 8n inclusive → counting; non-positive budget rejected) and verified numerically. `RuleBasedStrategySelector` is unchanged; the bandit's SMART path gets the same budget in the next item.
- **Bandit honours the SMART aux-memory budget** — `BanditStrategySelector` now accepts the same optional budget (new `BanditStrategySelector(long)` convenience plus a full 4-arg constructor; the no-arg and 3-arg constructors delegate with `Long.MAX_VALUE` = unlimited, so default behaviour is unchanged). Its `candidateArms` drops any arm whose declared `AuxMemory.estimatedBytes(n)` exceeds the budget — applied after capability gating and before the empty→introsort fallback — so under memory pressure the UCB search explores only in-place sorts: heap (`CONSTANT`) and intro / quick (`LOGARITHMIC`) survive while `LINEAR` merge / TimSort / counting / radix / learned drop out. Non-SMART policies still delegate to the rule-based base unchanged. Covered by `BanditSelectorTest` (a 256 KB cap at n=50k excludes the LINEAR comparison arms, and the LINEAR integer arms, even when they are made to look cheap; the default unlimited bandit still learns a LINEAR winner; non-positive budget rejected); arm filtering verified numerically.
- **`BeefSort.maxAuxMemory(long)` facade shortcut** — callers can cap SMART auxiliary memory without constructing a selector by hand: `BeefSort.…​maxAuxMemory(bytes)` installs a budgeted `CostModelStrategySelector(bytes)` through a new `resolveSelector()`, wired into both the `run()` / `feedInto()` paths and `adaptiveStream()`. - **Duplicate-tolerant `merge.wiki` (faithful WikiSort port)** — `WikiSortStrategy` is now a faithful, no-cache port of BonzaiThePenguin's WikiSort (the Kim&nbsp;&amp;&nbsp;Kutzner block merge), replacing the previous distinct-only fast path + rotation fallback. It handles duplicate keys **natively**, so near-distinct and duplicate-bearing inputs keep O(n&nbsp;log&nbsp;n) moves instead of degrading to the rotation merge's O(n&nbsp;log²&nbsp;n). Per merge level it pulls two √n *unique-value* buffers (one tags the A blocks for stable selection, one is swap scratch), rolls the A blocks through B dropping the minimum-headed block — ties broken by the tag ⇒ original order ⇒ stability — with buffer-backed seam merges, then redistributes; regions too low-cardinality to yield buffers fall back to the strictly-in-place rotation merge (`mergeInPlace`). The six allocating strategies report their actual footprint — merge ≈ 16n (working copy + scratch), TimSort ≈ 8n (reference copy), counting ≈ 16n + 4·range, LSD radix ≈ 32n + 4·radix, MSD radix ≈ 16n + 4·buckets, learned ≈ 28n — while the in-place sorts (insertion, heap, network, quick, intro, `merge.inplace`, `merge.wiki`) report 0. - **Memory-weighted bandit cost function** — `BanditStrategySelector.costWithMemory(bytesWeight)` returns a `ToDoubleFunction<SortResult>` = comparisons + moves + `bytesWeight`·`peakAuxBytes`, so the self-tuning bandit can learn from the measured aux (above): pass it as the `cost` argument. With a heavy weight the UCB converges away from scratch-heavy sorts (merge / TimSort / radix) toward the zero-aux in-place sorts; `bytesWeight = 0` reproduces the default comparisons + moves objective exactly. This closes the loop — capability → budgets → measured aux → learning that *uses* it. Covered by `BanditSelectorTest` (heavy weight + equal comparisons/moves but LINEAR-aux sorts reporting 16n / 8n scratch ⇒ an in-place winner; zero weight ⇒ identical to the default objective).

- **Soft graded aux-memory penalty (`CostModelStrategySelector`)** — the last open PROGRESS idea: a deterministic `λ · auxBytes` term for the SMART objective, alongside the existing hard over-budget filter. New `CostModelStrategySelector(long budget, double auxByteWeight)` ctor + `withAuxPenalty(double)` factory raise each SMART candidate's estimated cost by `auxByteWeight ×` its declared `AuxMemory.estimatedBytes(n)` (via a `penalizedCost` helper), so memory-hungry LINEAR sorts (merge/counting/radix/learned/TimSort) are discouraged in proportion to footprint rather than hard-cut, degrading toward the LOGARITHMIC introsort / CONSTANT in-place sorts as λ rises. `auxByteWeight = 0` (the default — both legacy ctors delegate to it) is byte-for-byte the original comparisons+moves objective: the penalty short-circuits off with no registry lookup. It composes with the hard budget (a candidate is excluded if over budget *and* penalized by its bytes) and is SMART-only (STABLE keeps its merge→WikiSort crossover). This is the deterministic, cost-model analog of `BanditStrategySelector.costWithMemory` (which tunes on *measured* `peakAuxBytes`). Covered by `CostModelSelectorTest` (tiny λ=0.001 keeps counting; heavy λ=3 flips counting→introsort, with the ~2.045/byte threshold verified numerically; a budget+penalty pair flips an in-budget counting to intro; negative/NaN λ rejected). Arithmetic validated in Python against the exact cost formulas; host-side `./gradlew build` is the gate.
- **Inversion-aware TimSort cost (galloping discount)** — the cost model's TimSort estimate now folds in the *global* inversion signal, closing the HANDOFF open item ("fold the inversion estimate into the TimSort run-count model"). The run count still comes from adjacency (the correct driver — a rotation / far-apart swap is only a couple of runs and genuinely cheap, which the inversion count would wrongly inflate), but the merge levels are scaled by a galloping factor `1 - GALLOP_DISCOUNT·globalOrder` where `globalOrder = min(sortednessRatio, 1 - inversionRatio)` — galloping credit only when *both* the local (adjacency) and global (inversion) signals agree the data is ordered. So two inputs with identical adjacency sortedness now score differently: a globally-ordered one (low inversions) gets the discount and can win as TimSort where the adjacency-only model picked introsort, while a locally-tidy-but-globally-disordered one (high inversions, e.g. far-apart swaps) is not over-credited. Safe by construction: `min(...)` caps the discount at the local sortedness, so a noisy/sampled inversion estimate can only *reduce* it (never inflate); and when no inversion count was measured the discount is skipped entirely, leaving the old adjacency-only estimate byte-identical. `GALLOP_DISCOUNT=0.4` is calibrated to preserve the random→introsort boundary. Covered by `CostModelSelectorTest` (same 0.70 adjacency: 10%-of-max inversions → TimSort, 70% → introsort) with all prior cost-model cases unchanged; the design and the "why a naive inversions-for-run-count swap is wrong" analysis are in `docs/adr-timsort-inversion-galloping.md`. Validated numerically in Python; host-side `./gradlew build` is the gate.

Records the integration-mechanism decision (optional SPI module over MRJAR / reflection), the need to bring the kernel to parity with the entropy-aware `RadixPlan`, and a benchmark-before-default gate. Implementation is host-side (needs Rust + JDK 22).
- **Phase 4 (Python intelligence) — design ready (`docs/adr-phase4-python-intelligence.md`).** Learned strategy selection behind the existing `StrategySelector` seam, reusing the `LearningStrategySelector.observe(...)` stream the engine already emits as a labeled training corpus. The ADR challenges the roadmap's "gRPC service" framing on hot-path-latency / availability grounds and leads with **offline training + in-process inference** (a `LearnedModelStrategySelector` wrapping the cost model, overriding only above a confidence margin + size gate), keeping the `SbsIntelligence` gRPC service as an optional Phase 4b for continual/fleet-wide learning behind a size gate + circuit breaker. First action item is just logging the `observe` corpus — which also measures whether a learned selector can beat the bandit at all before more is built.
- **Phase 5 (observability + scale) — design ready (`docs/adr-phase5-observability-scale.md`).** Two separable concerns: the substantive one is **out-of-core sort** — an external merge sort whose run-generation step reuses the existing in-memory engine (so every run still gets profile→select→sort), spilling sorted runs and k-way-merging them stably, memory-budgeted via the `maxAuxMemory` knob, able to feed CSRBT through the existing `StreamingFeeder`; distributed sort is deferred. The lighter one is **observability** — make the deliberately-deferred step-level `SortEvent` stream real (opt-in, zero cost when off) as a typed/versioned contract, and only then consider productizing the JS visualizer to TypeScript over it (the schema is the leverage, not the rewrite). With this, all three open roadmap phases (2, 4, 5) now have productization ADRs.
