- **WikiSort** — `WikiSortStrategy` (`merge.wiki`): O(n log n) stable in-place block merge, the successor to `merge.inplace` (O(n log² n)). Bottom-up + iterative (O(1) stack). Each merge takes a block-merge fast path on all-distinct regions — pull the √-sized largest-element internal buffer to the far right, align the A/B seam to the block grid, selection-sort blocks by head, repair seams with buffer-backed local merges, then insertion-sort the buffer in place (no re-merge: it already holds the largest values) — and falls back to the stable rotation merge for short or duplicate-bearing runs. Registered in `BuiltinStrategyProvider`, added to `DifferentialTest`, covered by `WikiSortTest`. Verified: matches the JDK reference across random + all pathological shapes, stable across key densities, and move count grows as O(n log n) (≈25% fewer moves than `merge.inplace` at n=1M, gap widening with n), at the cost of a larger comparison constant (~3.6 n log₂ n vs 1.3 — the usual block-merge trade).

- **WikiSort comparison-constant cut (threaded distinctness)** — the all-distinct gate is no longer recomputed from scratch at every merge. `WikiSortStrategy` threads a monotone `trustDistinct` flag (seeded by a one-time internal-duplicate scan of the insertion-sorted base runs): while it holds, every run produced so far is strictly increasing, so a merge decides distinctness with a single *cross-run-equal* walk (`crossRunEqual`, one comparison per element) instead of the two-comparison `fullyDistinct` (which also re-verifies within-run order). The first duplicate seen anywhere clears the flag, after which the exact, self-contained `fullyDistinct` gate is used — so the block-vs-rotation choice (hence the output, the move count, and stability) is provably identical to the old per-merge gate on every input; only the comparison count falls. - **Auxiliary-memory capability + memory-budgeted STABLE routing** — `StrategyCapabilities` now carries an `AuxMemory` growth class (`CONSTANT` / `LOGARITHMIC` / `LINEAR`) with `estimatedBytes(n)` (8 B/ref: `CONSTANT`→0, `LOGARITHMIC`→8·⌈log₂n⌉, `LINEAR`→8·n). Both `RuleBasedStrategySelector` and `CostModelStrategySelector` now derive the WikiSort `STABLE` crossover from an explicit 16 MB auxiliary-memory budget — `AuxMemory.LINEAR.estimatedBytes(size) >= AUX_MEMORY_BUDGET_BYTES` — instead of the magic `WIKI_MIN_SIZE = 2²¹`. - **Inversion-aware TimSort cost (galloping discount)** — the cost model's TimSort estimate now folds in the *global* inversion signal, closing the HANDOFF open item ("fold the inversion estimate into the TimSort run-count model"). The run count still comes from adjacency (the correct driver — a rotation / far-apart swap is only a couple of runs and genuinely cheap, which the inversion count would wrongly inflate), but the merge levels are scaled by a galloping factor `1 - GALLOP_DISCOUNT·globalOrder` where `globalOrder = min(sortednessRatio, 1 - inversionRatio)` — galloping credit only when *both* the local (adjacency) and global (inversion) signals agree the data is ordered. So two inputs with identical adjacency sortedness now score differently: a globally-ordered one (low inversions) gets the discount and can win as TimSort where the adjacency-only model picked introsort, while a locally-tidy-but-globally-disordered one (high inversions, e.g. far-apart swaps) is not over-credited. Safe by construction: `min(...)` caps the discount at the local sortedness, so a noisy/sampled inversion estimate can only *reduce* it (never inflate); and when no inversion count was measured the discount is skipped entirely, leaving the old adjacency-only estimate byte-identical. `GALLOP_DISCOUNT=0.4` is calibrated to preserve the random→introsort boundary. Covered by `CostModelSelectorTest` (same 0.70 adjacency: 10%-of-max inversions → TimSort, 70% → introsort) with all prior cost-model cases unchanged; the design and the "why a naive inversions-for-run-count swap is wrong" analysis are in `docs/adr-timsort-inversion-galloping.md`. Validated numerically in Python; host-side `./gradlew build` is the gate.

- **Phase 5 — External merge sort (ADR action items 1–4)** — `external/` package: `SpillSerializer<K>` interface + `forLongs/forIntegers/forStrings` built-ins; `SpillFile`/`SpillWriter`/`SpillReader` (temp-file lifecycle, `deleteOnExit` safety net for crash cleanup); `TournamentTree<K>` (min-heap k-way merge, run-index tiebreaker → stable between chunks); `ExternalMergeSorter<K>` (run generation by sorting fixed-size chunks with the full engine, multi-pass merge until ≤ `fanIn` runs remain, streaming output directly into a `CsrbtTarget` without materialising); `ExternalSortResult` metrics record. Facade: `BeefSort.external(SpillSerializer<K>)` returns an `ExternalSortBuilder` (non-static inner class; inherits comparator/keyEncoder/selector/policy/source) with `runSize(int)` + `fanIn(int)` + terminal methods `toList()` and `feedInto(OrderedSet<K>[, maxSize])`. - **Phase 2 — JMH benchmark results and ADR closed** — `RadixNativeBenchmark` (1 fork, 3 warmup + 5 measurement, 10s each) over n=[1k, 10k, 100k, 500k] on JDK 22.0.2, Windows 11, random integers in [0, 2n):

  | n | `radix.lsd` Java (µs/op) | `radix.lsd.rust` (µs/op) | Rust overhead |
  |---|---|---|---|
  | 1,000 | 10.7 ± 2.0 | 11.8 ± 0.9 | +10% |
  | 10,000 | 97.6 ± 7.4 | 115.4 ± 4.1 | +18% |
  | 100,000 | 1,021 ± 9 | 2,184 ± 326 | +114% |
  | 500,000 | 7,422 ± 145 | 15,340 ± 5,544 | +107% |

  **The native path is slower at every tested size.** FFM marshaling (off-heap `Arena`, two O(n) copy passes, permutation writeback) adds roughly one full sort's worth of work and does not amortize. Item 7 (selector integration) is deferred: no positive margin exists without **(a) rayon parallelism** in the kernel or **(b) an off-heap `SortBuffer`** that eliminates the copy passes. - **Phase 4 — Gate harness + corrected findings (ADR action item 2)** — `Phase4DecisionGate` runs a brute-force oracle (all applicable strategies, metered via `SortBuffer.compareValues` + `recordMove`) over 324 workloads. An initial run reported 0.00% regret because `JdkSortStrategy` used the raw `b.comparator()` (unmetered) and `b.set()` (no `recordMove`), so oracle cost was always 0 and the regret formula special-cased it. **Fixed `JdkSortStrategy`** to use `(x, y) -> b.compareValues(x, y)` and `b.recordMove()` in the writeback. After the fix the oracle sees 4 distinct winners across 324 workloads (oracle cost > 0 in all 324). Corrected results:

  | Selector | Exact match | Near-optimal (< 5% regret) | Mean regret | Max regret |
  |---|---|---|---|---|
  | `CostModelStrategySelector` | 196/324 (60.5%) | 196/324 (60.5%) | 386.52% | 6886.6% |
  | `BanditStrategySelector` | 212/324 (65.4%) | 236/324 (72.8%) | 191.94% | 7265.4% |

  **Real gaps exist**: reversed comparable-only (cost model picks `intro`; `jdk.timsort` detects the run in O(n) — up to 68× cheaper); clustered integer-keyed (cost model misses `counting`). Phase 4 items 3-5 remain open.
