- **Inversion-aware TimSort cost (galloping discount)** ‚Äî the cost model's TimSort estimate now folds in the *global* inversion signal, closing the HANDOFF open item ("fold the inversion estimate into the TimSort run-count model"). The run count still comes from adjacency (the correct driver ‚Äî a rotation / far-apart swap is only a couple of runs and genuinely cheap, which the inversion count would wrongly inflate), but the merge levels are scaled by a galloping factor `1 - GALLOP_DISCOUNT¬∑globalOrder` where `globalOrder = min(sortednessRatio, 1 - inversionRatio)` ‚Äî galloping credit only when *both* the local (adjacency) and global (inversion) signals agree the data is ordered. So two inputs with identical adjacency sortedness now score differently: a globally-ordered one (low inversions) gets the discount and can win as TimSort where the adjacency-only model picked introsort, while a locally-tidy-but-globally-disordered one (high inversions, e.g. far-apart swaps) is not over-credited. Safe by construction: `min(...)` caps the discount at the local sortedness, so a noisy/sampled inversion estimate can only *reduce* it (never inflate); and when no inversion count was measured the discount is skipped entirely, leaving the old adjacency-only estimate byte-identical. `GALLOP_DISCOUNT=0.4` is calibrated to preserve the random‚Üíintrosort boundary. - **Parallel Java LSD radix (`radix.lsd.parallel`)** ‚Äî the ADR-recommended successor to the rejected native kernel: a multi-threaded, stable LSD radix that captures the *one* lever that ever helped the Rust path (multicore) in pure Java, with none of its FFM marshaling, `unsafe`, JDK-22, or `long[]`-only constraints (`docs/adr-phase2-offheap-sortbuffer.md`). `ParallelRadixSortStrategy` reuses `RadixSortStrategy`'s entropy-aware key encoding (sign-flip + offset-by-min + `RadixPlan`) but parallelizes each counting pass with the classic **chunked-histogram ‚Üí disjoint-offset prefix ‚Üí parallel scatter** schedule: each thread counts its contiguous chunk into a private `count[chunk][digit]` row (no shared counters); one sequential prefix turns those rows into per-`(chunk,digit)` start offsets ordered **digit-major then chunk-major**; each thread then scatters its chunk *in order* into its own disjoint output ranges (no locks/atomics in the inner loop). That offset ordering is precisely what makes the parallel pass **stable** ‚Äî for a digit, chunk 0's run precedes chunk 1's, and the in-order scan preserves original order ‚Äî so the output is **byte-for-byte identical to sequential `radix.lsd` for any chunk count**, hence deterministic and core-count-independent. All `SortBuffer` access (the encode scan and the final metered write-back) stays single-threaded; the parallel work runs only on private primitive arrays, so the buffer's metering (moves = n) is unchanged and no concurrent access reaches it. Heeding the ADR's measured regression (the native rayon prototype was "catastrophic" until its radix was capped at 8 bits, because the per-pass `radix √ó chunks` histogram/offset matrix dominated), the multi-threaded path **caps bits-per-pass at 8** (256-entry histogram); the single-threaded path keeps the uncapped entropy plan, so below the `1<<16` element threshold it is byte-for-byte `radix.lsd`. Capabilities: stable + non-comparison + `parallel` + `requiresIntegerKeys` + `LINEAR` aux. Registered in `BuiltinStrategyProvider`; differential coverage added to `NonComparisonSortPropertyTest` (a forced 2-chunk arm, property-fuzzed against the JDK reference) plus a dedicated `ParallelRadixSortTest` (reference- and sequential-equality across 8 shapes √ó 11 sizes √ó 6 chunk counts, stability both forced-parallel and at scale, integer/long sign-flip extremes, and the 100k auto-parallel path). Algorithm verified in-sandbox by a faithful single-file fuzz harness (`PRadixCheck.java`, real `ForkJoinPool`): **972/972 cases pass** ‚Äî sorted order + permutation + stability across every shape/size/chunk-count, including full-range longs on the capped 8-bit path and heavy-tie inputs past the threshold. The host `./gradlew build` (JDK 17 + CSRBT) is the compile/test gate ‚Äî the sandbox has no `javac`. **Selector routing + JMH now added** ‚Äî see the next bullet.

- **Parallel radix ‚Äî JMH crossover sweep + provisional `SMART` routing (2026-06-23)** ‚Äî `bench/ParallelRadixBenchmark` is the head-to-head `radix.lsd` vs `radix.lsd.parallel` JMH sweep across n ‚àà {50k, 100k, 500k, 1M} (straddling the `1<<16` parallel-engage threshold), whose sole purpose is to locate the wall-clock crossover so routing is decided on measured data ‚Äî the same evidence-first rule applied to the native kernel. Ahead of that host run, `RuleBasedStrategySelector` (the engine default) now **routes** wide-range integer inputs under `SMART` to `radix.lsd.parallel` once `n >= PARALLEL_RADIX_CROSSOVER`, a named constant pinned to `ParallelRadixSortStrategy.PARALLEL_THRESHOLD` (`1<<16`, promoted to `public`). The branch sits *after* the counting gate (bounded-range integers still take `counting`) and is guarded on `registry.contains` (a registry without the parallel strategy falls back to `radix.lsd`); below the crossover the parallel strategy runs single-threaded anyway, so routing only engages where threads can actually help. Because `radix.lsd.parallel` is byte-for-byte identical to sequential `radix.lsd` (deterministic + stable, for any chunk count), this changes only wall-clock, never the result ‚Äî safe to ship before the benchmark, with the constant flagged **PROVISIONAL pending the host JMH crossover** (retune upward if measured higher than `1<<16`). The **cost-model and bandit selectors are deliberately untouched**: the cost model never picks `radix.lsd` to begin with (its `learned` ~5n arm dominates radix's ~8n, and parallel/sequential share an identical metered cost + `LINEAR` aux class, so a swap there would be dead code), and the bandit can't separate the two arms on metered cost ‚Äî the parallel advantage is purely wall-clock, which only the rule-based size gate expresses (rationale in `docs/adr-phase2-offheap-sortbuffer.md`). Covered by new `RuleBasedSelectorTest` cases (wide-range large integer ‚Üí `radix.lsd.parallel`; exactly at the crossover ‚Üí parallel; one element below ‚Üí `radix.lsd`; large *bounded*-range ‚Üí still `counting`, guarding the counting branch). Decision logic cross-checked in-sandbox by a faithful Python mirror of `smart()` (4 new + 4 regression cases all pass, plus a crossover sweep n=16‚Ä¶1M); host `./gradlew build` (JDK 17 + CSRBT) is the compile/test gate ‚Äî the sandbox has no `javac`. **JMH (2026-06-23, host JDK 22):** the first sweep read *inconclusive* ‚Äî but it was **confounded** (all-`par`-then-all-`seq`, `@Fork(1)`), exposed by a 50k *control* (sub-threshold ‚áí single-chunk ‚áí no threads) that still showed a ~14% gap. A **de-confounded re-run** (algorithm as fastest-varying `@Param` ‚áí seq/par back-to-back per n; 3 forks; n‚Üí5M; 50k control) is decisive and all-significant: `radix.lsd.parallel` is faster at **every** n ‚Äî raw **1.21√ó@100k ‚Üí 2.61√ó@5M** (1.06√ó‚Üí2.30√ó thread-only after netting the 14% baseline). So `PARALLEL_RADIX_CROSSOVER = 1<<16` is **VALIDATED** (not retuned ‚Äî already correct); routing tests unchanged. Full tables in the ADR.
